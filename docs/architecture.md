# 系统架构与关键取舍

## 业务链路

```text
企业资料 → 文件解析 → 企业画像抽取 → 冲突检测 → 确定性评分
                                      ↓
官方知识库 → 风险检索与引用 → 风险清单 → 90天计划 → 顾问审核 → PDF报告
```

## 组件职责

| 组件 | 职责 |
|---|---|
| Vue 3 | 项目创建、诊断结果、证据、冲突、审核、评测和监控 |
| FastAPI | 业务API、文件解析、评分、检索、报告和降级控制 |
| MySQL | 项目、文档、证据、评分、冲突、审核和运行记录的事实存储 |
| Redis | 最近任务状态；故障时自动回退MySQL |
| Dify | 可选的LLM企业画像抽取，要求字段携带原文证据 |
| N8N | Webhook触发诊断及外围自动化编排 |
| Nginx | 前端静态资源和 `/api` 反向代理 |

## 为什么不用复杂多Agent

项目核心是咨询辅助交付，优先级是证据、可解释性和稳定性。企业画像抽取适合LLM，评分、状态和质量门禁适合确定性代码。过早使用多Agent会增加调用成本、延迟和调试难度，却不会提高MVP的可信度。

## 为什么评分不用LLM

LLM只把非结构化资料映射为事实；Python规则引擎计算分数。相同输入得到相同分数，每一分都能追溯到字段、规则和证据，便于顾问复核。

## 三层降级

1. Dify异常：回退本地关键词抽取，并记录 `degraded`。
2. 知识检索无结果：标记 `insufficient`，强制人工确认。
3. Redis异常：最近任务状态回退MySQL，不阻断主链路。

## 数据可信机制

- 资料缺失不等于企业不合规；
- Dify字段没有原文证据时不会进入画像；
- 肯定与否定事实冲突时，字段从评分中隔离；
- 专业风险附带官方来源；
- 最终报告需要顾问审核。
